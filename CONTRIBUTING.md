# Contributing to permit-golang

Thank you for helping improve the Permit.io Go SDK. This guide covers the
minimum workflow for small fixes, SDK wrapper changes, tests, and generated API
code.

## Local Setup

Install Go 1.19 or newer, then download the module dependencies:

```bash
go mod download
```

Before opening a pull request, run the same checks used by CI for normal code
changes:

```bash
go fmt ./...
go build ./...
go test ./...
```

Some integration tests expect Permit.io service configuration such as
`PDP_API_KEY`, `PDP_URL`, `API_URL`, `PROJECT`, and `ENV`. If you cannot run
those locally, say so in the pull request and include the checks you did run.

## Choosing a Change

Small, focused pull requests are easiest to review. Good starter changes include:

- fixing SDK wrapper behavior in `pkg/api`, `pkg/enforcement`, `pkg/config`, or
  `pkg/errors`
- adding tests for an existing method or edge case
- clarifying README examples
- wiring an already generated model or API method into the handwritten SDK layer

For larger API-surface changes, open or comment on an issue first. Include the
Permit API endpoint, the expected SDK behavior, and how the change affects
downstream integrations such as Terraform providers.

## Generated API Code

Files under `pkg/openapi` are generated by OpenAPI Generator and include a
`DO NOT EDIT` notice. Treat generated files differently from handwritten SDK
wrappers:

- Prefer changing handwritten wrappers and tests when the generated client
  already exposes the needed API shape.
- If a generated model or endpoint appears to be missing or stale, explain the
  source API mismatch in the issue or pull request before editing generated
  files manually.
- Keep generated-file diffs as small as possible, and avoid mixing generator
  churn with unrelated SDK behavior changes.
- If maintainers provide generator configuration, templates, or an updated
  OpenAPI spec for the change, use those sources instead of hand-editing output.

This repository does not currently include a checked-in generator config or
template directory. If your fix requires one, ask maintainers for the expected
generation path before opening a broad generated-code pull request.

## Pull Request Checklist

Before requesting review:

- keep the change focused on one issue or behavior
- run `go fmt ./...`
- run `go build ./...`
- run `go test ./...`, or explain which integration tests could not run locally
- add or update tests for behavior changes when practical
- document any manual generated-code edits and the API source they came from

Clear scope and verification notes help maintainers review the SDK faster.
